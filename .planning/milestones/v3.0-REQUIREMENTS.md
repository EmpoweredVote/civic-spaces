# Requirements: Civic Spaces

**Defined:** 2026-04-05
**Milestone:** v3.0 — UI/UX Redesign
**Core Value:** Every Connected user is part of four geographic communities plus specialized civic spaces — and they can move fluidly between all of them from a single hub.

## v3.0 Requirements

### Hero Banner

- [x] **HERO-01**: User sees a full-width hero banner at the top of each slice tab — geo photo with slice name, tagline, pill badges (level, jurisdiction, member count, slice number), and a 2-sentence description overlaid
- [x] **HERO-02**: Hero photo changes when user switches slice tabs
- [x] **HERO-03**: Active slice tab is visually highlighted (brand teal) in the tab bar

### Layout

- [x] **LAYOUT-01**: On desktop, slice content uses a two-column layout — feed (~65%) left, sidebar (~35%) right
- [x] **LAYOUT-02**: On mobile, sidebar is hidden and feed is full-width single column

### Sidebar

- [x] **SIDE-01**: If user is authenticated with the Empower Compass (cs_token available), they see a live radar chart of their calibrated civic compass in the sidebar
- [x] **SIDE-02**: If cs_token is unavailable or compass is uncalibrated (fewer than 3 answered topics), the widget shows a prompt card: "Set up your Civic Compass" with a "Calibrate Now" link to compassv2.empowered.vote/results
- [x] **SIDE-03**: User sees a 'Representing This Community' widget listing elected officials relevant to this slice's jurisdiction level
- [x] **SIDE-04**: Each rep card shows name, title; NO party badge (anti-partisan policy); widget absent entirely when no reps found
- [x] **SIDE-05**: User sees a 'Tools for This Community' widget with icon cards for confirmed-live EV ecosystem tools; non-live tools excluded entirely

### Photos

- [x] **PHOTO-01**: Jurisdiction-specific hero photos served via CDN URL (Wikipedia REST API; photo_url DB override path also active for future curated imagery)
- [x] **PHOTO-02**: Each slice type (Neighborhood, District, State, Federal, Unified, Volunteer) has at least one curated hero photo; Bloomington-pilot slices get location-specific imagery

### Cleanup

- [x] **CLEAN-01**: `friendCount` dead prop removed from `MutualFriendsList`
- [x] **CLEAN-02**: `onNavigateToThread` vestigial prop removed from `NotificationListProps`

## Future Requirements (v4.0+)

- **COMP-01**: Issue Alignment Compass compares user priorities vs. a specific rep's positions (side-by-side overlay)
- **COMP-02**: User can open a full-page compass comparison view

### Visual Polish

- **VIS-01**: Animated hero banner transitions on tab switch
- **VIS-02**: Skeleton loading states for sidebar widgets

## Out of Scope

| Feature | Reason |
|---------|--------|
| Real-time rep data sync | Accounts API owns rep data; Civic Spaces is a consumer — no local rep DB |
| User-uploadable slice photos | Moderation complexity; curated imagery is sufficient |
| Debates / Symposiums / Announcements tabs | Future feature layers, explicitly deferred in v2.0 |
| Focus Communities | Future hub expansion |
| Algorithmic feed ranking | Conscious anti-feature — chronological feed is a civic legitimacy signal |

## Milestone Summary

**Shipped:** 14 of 14 v3.0 requirements

**Adjusted during implementation:**
- SIDE-04: Original spec said "party badge" — anti-partisan policy determined party affiliation must never appear; rep cards show name + title only. Stronger than the requirement.
- SIDE-05: Original spec said "gracefully degraded" for non-live tools. Approach changed to complete exclusion — no coming-soon cards, only confirmed-live tools.
- PHOTO-01/02: Approach pivoted from Supabase Storage uploads to Wikipedia REST API. Outcome stronger (automatic geo-appropriate photos for all US states/counties, no uploads needed).

**Dropped:** None
