# Milestone v2.0: All Slices

**Status:** ✅ SHIPPED 2026-04-04
**Phases:** 6–8
**Total Plans:** 11

## Overview

Expand from Federal-only to all slice types — activating the remaining geo slices, launching the Unified and Volunteer slices, and giving users a profile page that anchors their civic identity.

## Phases

### Phase 6: Hub Expansion

**Goal**: The hub has its permanent two-column layout, geo slices (N/L/S) are fully active forums, scroll state is independently preserved per tab, and notifications route users to the correct slice tab.
**Depends on**: Phase 5
**Requirements**: HUB-06, HUB-08, SLCE-02, SLCE-03
**Plans**: 5 plans

Plans:

- [x] 06-01-PLAN.md — Hub layout redesign + useAllSlices hook + activate all 4 geo slice feeds + Unified/Volunteer disabled shells
- [x] 06-02-PLAN.md — Per-tab scroll state — independent scroll position preservation across all hub tabs
- [x] 06-03-PLAN.md — Notification routing — resolve post slice_id to owning tab, navigate to correct tab on notification tap
- [x] 06-04-PLAN.md — Gap closure — TS errors (TS2741/TS6133/TS7053) + mobile notification routing missing prop
- [x] 06-05-PLAN.md — TypeScript gap closure — zero-error tsc build (16 pre-existing errors across 7 files)

**Details:**

All 4 geo slice feeds (N/L/S/F) are simultaneously mounted via CSS hidden pattern — preserving React Query subscriptions, realtime connections, and DOM scroll position when the user switches tabs. Per-tab state maps (activePostIds, scrollToLatest) prevent cross-tab thread view interference. Notification routing resolves `posts.slice_id` to a `TabKey` and switches to the correct geo tab before opening the thread, with a 'federal' fallback on any error. Desktop and mobile notification lists both carry the new `onNavigateToSliceThread` prop.

### Phase 7: New Slice Types

**Goal**: The Unified and Volunteer slices exist in the schema, the assignment service populates them automatically, and their hub tabs are fully active forums — Volunteer tab is visible only to users with the Volunteer role.
**Depends on**: Phase 6
**Requirements**: HUB-07, SLCE-01, UNIF-01, UNIF-02, UNIF-03, UNIF-04, VOL-01, VOL-02, VOL-03, VOL-04, VOL-05
**Plans**: 3 plans

Plans:

- [x] 07-01-PLAN.md — Schema migration (CHECK constraint extension for unified/volunteer) + TypeScript type updates (SliceType, SliceInfo)
- [x] 07-02-PLAN.md — Slice assignment service extension (Unified check-before-insert, Volunteer role check via POST /api/roles/check, revocation DELETE, route wiring)
- [x] 07-03-PLAN.md — Frontend activation (useAllSlices extension, AppShell/SliceTabBar overhaul, Volunteer tab conditional rendering, feed headers)

**Details:**

The `slices` CHECK constraint was extended from 4 to 6 values (added `'unified'`, `'volunteer'`). SliceType TypeScript union updated in sync. The assignment service now runs Unified assignment before jurisdiction check (non-geo slice, users without jurisdiction still get Unified), uses check-before-insert for stable cohort semantics, and checks the Volunteer role via `POST /api/roles/check` with `{ feature_scope: 'volunteer', jurisdiction_geoid: null }`. Volunteer role is re-verified and revoked on every login. The frontend uses CSS-hidden panel mounting for all 5 geo+special feeds; Volunteer tab is conditionally absent from DOM (not just hidden) for non-Volunteer users.

### Phase 8: Profile Pages

**Goal**: Any user's display name is tappable throughout the app and opens a profile page that shows their civic identity — slice memberships, activity counts, and social connections appropriate to the viewer's relationship with them.
**Depends on**: Phase 7
**Requirements**: PROF-01, PROF-02, PROF-03, PROF-04, PROF-05, PROF-06, PROF-07
**Plans**: 3 plans

Plans:

- [x] 08-01-PLAN.md — Database RPCs (get_profile_stats, get_mutual_friends), wouter install, profile hooks (useProfileStats, useProfileSlices, useMutualFriends), extend useProfileById
- [x] 08-02-PLAN.md — ProfilePage component tree (header, stats strip, slices, friends, skeleton) + App.tsx Router integration with CSS-hidden AppShell
- [x] 08-03-PLAN.md — Display name tap navigation wiring (PostCard, ReplyCard, NotificationItem, FriendsList) + AppShell cleanup (remove UserProfileCard overlay)

**Details:**

Two new Supabase RPCs: `get_profile_stats` returns a JSON object with `post_count`, `reply_count`, and `friend_count`; `get_mutual_friends` returns mutual friends between any two users. Wouter installed for SPA routing; ProfilePage is a fixed `inset-0` overlay that stacks above the CSS-hidden AppShell without layout coupling. Display name taps from 8 surfaces (PostCard, ReplyCard, ThreadView, NotificationItem, MemberDirectory, FriendsList, FriendRequestButton, UserProfileCard) all navigate to `/profile/:userId`. Own-view shows full friends list; other-view shows mutual friends only with no total count disclosure.

## Milestone Summary

**Key Decisions:**

- CSS hidden used (not conditional rendering) for all geo SliceFeedPanels — preserves React Query cache, realtime subscriptions, and DOM scroll across tab switches
- check-before-insert used for Unified (not upsert) — stable 2-year cohort semantics; users stay in same sibling slice
- Unified assignment runs before jurisdiction check — non-geo slice, Connected users without jurisdiction still get Unified
- Volunteer revocation on every login when role absent — prompt removal without webhook dependency (90s server-side cache means revocation lags up to 90s by design)
- ProfilePage as fixed `inset-0` overlay — stacks above CSS-hidden AppShell without layout coupling
- Shared slice detection by `slice_type` key, not `slice_id` — sibling slices differ by ID but same type = same geo tier
- `get_profile_stats` returns JSON (not TABLE) — aggregates three scalar counts into one ergonomic object
- `onAuthorTap` callback pattern fully removed — leaf components own navigation via `useLocation`; no prop threading through SliceFeedPanel
- `NoJurisdictionBanner` gated on `!hasJurisdiction && !slices['unified']` — Unified users without geo slices see feeds, not banner
- `SliceType` now includes `'volunteer'` — resolves Phase 6 deferral; Phase 6 cast `(activeTab as SliceType)` is now type-safe

**Issues Resolved:**

- 16 pre-existing TypeScript errors from the v1.0 codebase (Phase 6 gap closure)
- Mobile notification routing gap — mobile Sheet NotificationList was missing `onNavigateToSliceThread` prop (closed by quick task 001)
- Total friend count disclosure on other-user profile (gap fix in Phase 8)
- MemberDirectory was still using UserProfileCard overlay instead of profile navigation (gap fix in Phase 8)

**Issues Deferred:**

- `MutualFriendsList` accepts `friendCount` prop but never renders it — dead prop surface (harmless; cleanup in future milestone)
- `onNavigateToThread` is vestigial in `NotificationListProps` — always passed, never invoked; cleanup in future milestone

**Technical Debt Incurred:**

- None beyond the two dead props noted above, both documented in the v2.0 audit
