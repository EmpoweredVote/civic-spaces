  <action>
Create migration file with the following sections, in order:

1. **Notifications table** in `civic_spaces` schema:
   - `id uuid PRIMARY KEY DEFAULT gen_random_uuid()`
   - `recipient_id text NOT NULL` (auth.jwt() ->> 'sub' pattern)
   - `event_type text NOT NULL CHECK (event_type IN ('reply', 'friend_request', 'friend_accepted'))`
   - `actor_id text NOT NULL` (first/primary actor who triggered the notification)
   - `actor_ids text[] NOT NULL DEFAULT '{}'` (array of all actor user_ids for grouped display)
   - `reference_id text NOT NULL` (post_id for replies, user_id for friend events)
   - `reference_excerpt text` (denormalized post body excerpt at insert time, NULL for friend events)
   - `event_count integer NOT NULL DEFAULT 1`
   - `is_read boolean NOT NULL DEFAULT false`
   - `group_window date NOT NULL DEFAULT CURRENT_DATE` (24-hour collapsing key)
   - `created_at timestamptz NOT NULL DEFAULT now()`
   - `updated_at timestamptz NOT NULL DEFAULT now()`
   - Index on `(recipient_id, is_read, updated_at DESC)` for the main query path
   - Partial unique index: `CREATE UNIQUE INDEX notifications_group_idx ON civic_spaces.notifications (recipient_id, event_type, reference_id, group_window) WHERE is_read = false`

2. **RLS policies** (following established project pattern with `(SELECT auth.jwt() ->> 'sub')` subquery wrapping):
   - `ALTER TABLE civic_spaces.notifications ENABLE ROW LEVEL SECURITY`
   - SELECT policy: `recipient_id = (SELECT auth.jwt() ->> 'sub')` for `authenticated`
   - UPDATE policy: `recipient_id = (SELECT auth.jwt() ->> 'sub')` for `authenticated`
   - No INSERT or DELETE from client — triggers handle inserts

3. **Trigger function `civic_spaces.notify_on_reply()`** — AFTER INSERT ON `civic_spaces.replies`:
   - Look up post owner: `SELECT user_id INTO v_post_owner FROM civic_spaces.posts WHERE id = NEW.post_id AND NOT is_deleted`
   - Also check if reply is to another reply: if `NEW.parent_reply_id IS NOT NULL`, look up that reply's `user_id` as `v_reply_owner`
   - Skip self-notification (if post/reply owner = NEW.user_id)
   - Notify post owner (always, unless self):
     - Get excerpt: `SELECT LEFT(body, 50) INTO v_excerpt FROM civic_spaces.posts WHERE id = NEW.post_id`
     - UPSERT into notifications with `ON CONFLICT (recipient_id, event_type, reference_id, group_window) WHERE is_read = false DO UPDATE SET actor_ids = CASE WHEN NEW.user_id = ANY(notifications.actor_ids) THEN notifications.actor_ids ELSE array_append(notifications.actor_ids, NEW.user_id) END, event_count = notifications.event_count + 1, updated_at = now()`
   - If `v_reply_owner` exists and is different from post owner and different from NEW.user_id, also notify the reply owner (separate upsert, same pattern)
   - Use `NEW.user_id` for actor identification (not auth.jwt() — triggers use row data)

4. **Trigger function `civic_spaces.notify_on_friendship_change()`** — AFTER INSERT OR UPDATE ON `civic_spaces.friendships`:
   - On INSERT (friend request sent):
     - If `NEW.status = 'REQ_LOW'`: recipient is `NEW.user_high`, actor is `NEW.user_low`
     - If `NEW.status = 'REQ_HIGH'`: recipient is `NEW.user_low`, actor is `NEW.user_high`
     - UPSERT into notifications with event_type 'friend_request', reference_id = actor user_id
   - On UPDATE where `NEW.status = 'FRIEND' AND OLD.status != 'FRIEND'` (acceptance):
     - If `OLD.status = 'REQ_LOW'`: notify `NEW.user_low` (original requester) that `NEW.user_high` accepted
     - If `OLD.status = 'REQ_HIGH'`: notify `NEW.user_high` that `NEW.user_low` accepted
     - event_type 'friend_accepted', reference_id = acceptor user_id
   - reference_excerpt = NULL for all friend events

5. **Create triggers:**
   - `CREATE TRIGGER reply_notification AFTER INSERT ON civic_spaces.replies FOR EACH ROW EXECUTE FUNCTION civic_spaces.notify_on_reply()`
   - `CREATE TRIGGER friendship_notification AFTER INSERT OR UPDATE ON civic_spaces.friendships FOR EACH ROW EXECUTE FUNCTION civic_spaces.notify_on_friendship_change()`

6. **Public schema view** (following established pattern from Phase 3):
   - `CREATE OR REPLACE VIEW public.notifications AS SELECT * FROM civic_spaces.notifications`
   - `GRANT SELECT, UPDATE ON public.notifications TO authenticated`

7. **Realtime publication** (required for private schema tables):
   - `GRANT SELECT ON civic_spaces.notifications TO authenticated`
   - `ALTER PUBLICATION supabase_realtime ADD TABLE civic_spaces.notifications`

IMPORTANT: All trigger functions use `NEW.user_id`, `NEW.user_low`, `NEW.user_high` from the triggering row — never `auth.uid()` or `auth.jwt()`. Only RLS policies use `auth.jwt() ->> 'sub'`.
